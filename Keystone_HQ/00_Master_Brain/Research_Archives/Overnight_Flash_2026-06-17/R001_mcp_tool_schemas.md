# Best Practices for Model Context Protocol (MCP) Tool Schemas in 2026: Architecting for Agentic Reliability

## Summary
* **Stateless Architecture Execution (SEP-2575):** The protocol has transitioned to a fully stateless model. Standard session handshakes have been removed; servers now use explicit, opaque state handles passed as tool arguments to maintain state across invocations.
* **JSON Schema 2020-12 Enforced (SEP-2106):** Schemas are strictly validated at the client layer using the 2020-12 specification, eliminating legacy nullable workarounds and preventing invalid arguments from reaching the server.
* **Separation of Documentation (SEP-1382):** Tool descriptions (`tool.description`) focus solely on selection intent and exclusionary guidelines, while specific parameter formatting rules are strictly housed within nested `inputSchema` descriptions.
* **LLM-Centric Error Recovery via `isError: true`:** Operational, parameter, or business logic failures must return a successful protocol response with `isError: true` and clear, actionable feedback to allow self-correction, reserving hard transport errors for malformed RPCs.
* **Code Mode and Progressive Discovery:** Implementing progressive tool discovery reduces context bloat, while "Code Mode" compiles tools into a sandboxed environment (Deno, WebAssembly, Monty) so models can execute complex logic without round-trip overhead.

## Key Findings

### The 2026 Architectural Foundations of MCP
The Model Context Protocol has evolved to support high-throughput, autonomous enterprise agent fleets by shifting from a connection-oriented stateful system to a stateless design. Under the 2026 standards:
* **Statelessness (SEP-2575):** Deprecated the `initialize` / `notifications/initialized` handshake. Requests now route statelessly based on HTTP headers `Mcp-Method` and `Mcp-Name` at the API Gateway level. State tracking is done using opaque, cryptographically random handles (e.g., `bsk_a1b2c3` for a shopping cart) generated by the server and returned to the agent, which passes it as a required parameter in subsequent calls.
* **Dynamic TTL Caching (SEP-2549):** Tool lists now feature `ttlMs` and `cacheScope` ("public" or "private") to allow caching at the client or edge proxy, dramatically reducing discovery overhead.
* **Header Mapping (SEP-2243):** Allows specific tool parameters annotated with `x-mcp-header` to map directly to HTTP headers. This enables secure edge routing and authentication without parsing body JSON.

### Schema Modernization: JSON Schema 2020-12 (SEP-2106)
Full conformance to JSON Schema 2020-12 is now mandatory. This allows developers to use advanced schema composition keywords like `oneOf` or `anyOf`, write complex validation rules, and resolve `$ref` declarations directly at the client layer. Hallucinated or malformed arguments are rejected before they touch the server network.

### Documentation Separation and Intent-Based Architecture
* **Documentation Dichotomy (SEP-1382):** 
  * `tool.description` is for discovery. It must be a high-level conceptual explanation of what the tool does, when to use it, and what it does *not* do (exclusionary guidance).
  * `inputSchema.properties.description` is for invocation. It contains exact parameter rules, valid ranges, regex boundaries, and dependency relationships.
* **Intent-Based vs. REST Mapping:** Mapping REST endpoints 1:1 to tools is a major anti-pattern causing context bloat. Instead, design cohesive, task-oriented tools (e.g., `fetch_customer_context` instead of three separate endpoint tools). Names should follow the "Verb + Resource" pattern within the first five words for optimal vector index retrieval.

### Enterprise Schema Governance
Centralized validation using frameworks like Pydantic or Zod ensures schema standardization. Enums should be used for finite states (e.g., status flags) but *never* for dynamic database fields or IDs. Output validation schemas are required to prevent unstructured content arrays, which force the use of expensive extraction helper models.

### LLM-Centric Error Handling and Recovery
Hard exceptions terminate the agent loop. Therefore, all business logic or parameter errors must return a successful transport response with `isError: true`. The text output should describe the violated constraint, the invalid value passed, and how to correct it. This allows the model to reason, update its strategy, and retry autonomously.

| Component Requirement | Standardized 2026 Best Practice Implementation | Deprecated or Suboptimal Approach |
| :--- | :--- | :--- |
| **Tool Naming** | Intent-based, grouped logic (e.g., `fetch_customer_context`) | 1:1 API mapping (e.g., `get_user_api_v2_endpoint`) |
| **Root Description** | High-level intent, side-effects, latency, and exclusionary guidance | Parameter formatting rules, complex JSON mapping examples |
| **Input Property Definitions** | JSON Schema 2020-12, bounded ranges, regex validation, explicit defaults | Unbounded string types, generic object types without property definitions |
| **Enum Utilization** | Finite state indicators (e.g., `"status": ["pending", "active"]`) | Dynamic database IDs, free-form text inputs, or arbitrary floating numbers |
| **State Tracking** | Explicit, opaque strings (e.g., `bsk_a1b2`) passed continuously between calls | Implicit session tracking via connection persistence or sticky IP routing |
| **Schema Generation** | Pydantic or Zod models strictly enforcing type coercion | Ad-hoc Python docstrings or manually written, unvalidated JSON blocks |

### Advanced Client Integration Patterns
* **Progressive Tool Discovery:** To avoid bloating the context window, the client initially exposes a single `search_tools` tool. Once the agent identifies potential tools, it calls `get_tool_details` to retrieve the schemas of only the chosen tools.
* **Code Mode (Programmatic Tool Calling):** Instead of sequential round-trips (Agent -> Tool A -> Agent -> Tool B), the agent writes a sandboxed script (Deno, Monty, WebAssembly) that calls tools via a local programmatic API compiled from active schemas. The script runs in isolation, and the agent only processes the final script results.

## Action Items
1. **Audit and Upgrade Schemas:** Refactor all MCP server `inputSchema` and `outputSchema` definitions to strictly adhere to JSON Schema 2020-12 (SEP-2106).
2. **Refactor Tool Descriptions:** Move all parameter formatting rules out of the root tool descriptions and into the property-level descriptions (SEP-1382). Add explicit exclusionary guidance at the root level.
3. **Implement `isError: true` Error Handling:** Replace exceptions thrown at the protocol level with successful response objects containing `isError: true` and detailed corrective text.
4. **Deploy Sandboxed Code Execution:** Prepare a sandboxed execution runtime (Deno or WebAssembly-based) to support "Code Mode" programmatic tool calling.
5. **Optimize Gateways for Stateless Routing:** Leverage `Mcp-Method` and `Mcp-Name` HTTP headers for routing and implement edge-caching with `ttlMs`.
